Documentation

Everything you need to set up, extend, and operate Tapcraft OS.

Installation

Tapcraft requires Docker, Docker Compose, and Node.js 18+ (for the UI dev server).

terminal
$ git clone https://github.com/tapcraft-io/tapcraft-os.git
$ cd tapcraft-os
$ cp .env.example .env

Edit .env if you need to set a specific API key or encryption key. Leaving them blank causes auto-generation on first startup.

First Run

Start all three backend services (Temporal, API, Worker) with Docker Compose:

terminal
$ docker compose up --build

Then start the UI development server:

terminal
$ cd ui && npm install && npm run dev
ServiceURL
APIhttp://localhost:8001
UIhttp://localhost:5180
Temporal UIhttp://localhost:8082

On first startup the API auto-generates a TAPCRAFT_API_KEY and prints it to the console. Use this key to log in to the web UI.

Workspaces

A workspace is the top-level container for all your activities, workflows, schedules, and runs. Each workspace maps to a directory under /app/workspace/workspace_{id}/.

A workspace can optionally be connected to a git repository. When you sync, Tapcraft clones the repo into the workspace and auto-discovers any activities and workflows it finds.

workspace directory layout
workspace/
└── workspace_1/
    ├── workflows/          # Deployed or generated workflows
    ├── activities/         # User-defined activities
    ├── apps/               # App code
    └── repo/               # Cloned git repository
        ├── workflows/
        ├── activities/
        └── ...

Activities

An activity is a reusable unit of work — an HTTP call, a database query, a file parse, or any I/O operation. Activities are the only place where side effects should happen; workflows orchestrate them but never perform I/O directly.

Tapcraft ships with four built-in activities:

NameDescription
net.http.parallelExecute multiple HTTP requests concurrently
feed.rss.readFetch and parse an RSS/Atom feed
data.parse_xmlParse an XML string into a nested dict
data.dedupDeduplicate a list of dicts by a key field

You can also write your own — see Writing Activities.

Workflows

A workflow is a Temporal workflow class that orchestrates one or more activities into a durable pipeline. Workflows survive process restarts, have built-in retries, and maintain full execution history in Temporal.

Workflows can be created in two ways:

  • Visual editor — build a DAG in the UI; Tapcraft generates Python code from the graph
  • Code-first — write Python directly and deploy via MCP or the git sync

See Writing Workflows for the full guide.

Schedules

A schedule binds a workflow to a cron expression. Tapcraft creates a native Temporal schedule under the hood, giving you timezone support, pause/resume, and execution history.

Cron expressions use standard 5-field format: minute hour day-of-month month day-of-week.

ExpressionMeaning
0 9 * * 1-59:00 AM every weekday
*/15 * * * *Every 15 minutes
0 0 1 * *Midnight on the 1st of every month
@dailyOnce a day at midnight
@hourlyOnce an hour at :00

Secrets

Secrets store API keys, tokens, and credentials encrypted at rest using Fernet symmetric encryption. The encryption key is set via TAPCRAFT_SECRET_KEY.

Activities retrieve secrets at runtime:

my_activity.py
from src.services.secrets import get_secret

@activity.defn(name="my.activity")
async def my_activity(config: dict) -> dict:
    api_key = await get_secret("OPENAI_API_KEY")
    # use api_key in HTTP calls...

Webhooks

Webhooks let external services trigger workflow executions via HTTP. Each webhook exposes a unique inbound URL that accepts POST requests without authentication, making it easy to wire up GitHub, Stripe, Slack, or any system that supports outgoing webhooks.

How it works

  1. Create a webhook via the API or MCP, specifying which workflow to trigger and an optional secret for HMAC verification.
  2. The webhook is assigned a unique path. The inbound URL is POST /hooks/{path}.
  3. When the inbound URL receives a request, Tapcraft verifies the signature (if configured), then starts the linked workflow with the request body as input.

HMAC Verification

If a webhook has a secret, incoming requests must include an X-Webhook-Signature header. Tapcraft computes an HMAC-SHA256 digest of the raw request body using the webhook's secret and compares it to the header value. Requests with missing or invalid signatures are rejected with 401.

terminal
# Sending a signed webhook request
$ BODY='{"event": "deploy", "ref": "main"}'
$ SIG=$(echo -n "$BODY" | openssl dgst -sha256 -hmac "your-webhook-secret" | cut -d' ' -f2)
$ curl -X POST http://localhost:8001/hooks/my-deploy-hook \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Signature: $SIG" \
  -d "$BODY"

OAuth Credential Manager

The OAuth Credential Manager provides full OAuth 2.0 provider and credential management. Register OAuth providers (Google, GitHub, Slack, etc.), run the authorization flow, and store access/refresh tokens — all encrypted at rest with Fernet symmetric encryption.

How it works

  1. Register a provider — store the client ID, client secret, authorization URL, token URL, and scopes.
  2. Authorize — call the authorize endpoint to get a redirect URL. The user completes the OAuth flow in their browser.
  3. Callback — exchange the authorization code for access and refresh tokens, which are stored encrypted.
  4. Use in activities — activities retrieve the credential at runtime and use the access token for API calls.
  5. Refresh — when tokens expire, call the refresh endpoint to get new ones.

Security

  • All client secrets and tokens are encrypted at rest using the TAPCRAFT_SECRET_KEY Fernet key.
  • Access tokens and refresh tokens are never exposed in plaintext via the API — only metadata is returned in list responses.
  • Provider secrets are encrypted on creation and decrypted only when needed for token exchange.

Writing Activities

An activity is an async function decorated with @activity.defn. It receives a single dict argument and returns a dict.

activities/my_activity.py
from temporalio import activity
from typing import Any, Dict
import httpx

@activity.defn(name="my_namespace.fetch_data")
async def fetch_data(config: Dict[str, Any]) -> Dict[str, Any]:
    """Fetch data from an external API."""
    url = config.get("url", "https://api.example.com/data")
    timeout = config.get("timeout", 30)

    async with httpx.AsyncClient() as client:
        resp = await client.get(url, timeout=timeout)
        resp.raise_for_status()

    return {"data": resp.json(), "status": resp.status_code}

Rules

  • Activities may perform I/O — HTTP calls, file reads, database queries, etc.
  • Receive a single dict config and return a dict result.
  • Give each activity a unique name using dot-separated namespaces.
  • Activities can access encrypted secrets via await get_secret("NAME").

Discovery

Place your file in one of these directories and the worker discovers it automatically:

  • workspace/workspace_{id}/activities/
  • workspace/workspace_{id}/repo/activities/
  • workspace/workspace_{id}/repo/{subdir}/activities/ (monorepo)

The worker scans for new files every 5 seconds. No restart needed.

Writing Workflows

A workflow is a class decorated with @workflow.defn containing a @workflow.run async method.

workflows/daily_report.py
from datetime import timedelta
from temporalio import workflow
from temporalio.common import RetryPolicy

@workflow.defn
class DailyReportWorkflow:
    """Fetch metrics and send a summary report."""

    @workflow.run
    async def run(self, config: dict) -> dict:
        # Step 1: fetch metrics
        metrics = await workflow.execute_activity(
            "metrics.pull",
            {"days": config.get("days", 7)},
            start_to_close_timeout=timedelta(minutes=2),
            retry_policy=RetryPolicy(maximum_attempts=3),
        )

        # Step 2: wait before sending
        await workflow.sleep(5)

        # Step 3: send report
        result = await workflow.execute_activity(
            "notifications.send_report",
            {"metrics": metrics, "channel": "#team"},
            start_to_close_timeout=timedelta(seconds=30),
        )

        workflow.logger.info("Report sent successfully")
        return {"status": "done", "items": len(metrics)}

Key Rules

Important: Workflow code must be deterministic. Never perform I/O, use datetime.now(), generate random numbers, or access the filesystem directly. All side effects must go through activities.
  • Use workflow.execute_activity("name", config_dict, start_to_close_timeout=...) to call activities.
  • Always set start_to_close_timeout — it is required.
  • Use await workflow.sleep(seconds) for delays (not asyncio.sleep).
  • Use workflow.logger for logging (not the logging module).
  • Accept config: dict and use .get() with defaults.
  • Return a dict summarizing the result.

Timeout Guidelines

Activity typeRecommended timeout
Fast operations (parsing, formatting)timedelta(seconds=30)
API callstimedelta(minutes=2-5)
Long-running operationstimedelta(minutes=10+)

Discovery

Workflows are discovered from the same paths as activities, but in workflows/ directories. The worker watches for new files every 5 seconds.

Error Handling

Every activity node in a workflow graph supports configurable retry policies and timeouts. These are set as underscore-prefixed keys in the node's config and are translated to Temporal RetryPolicy and timeout arguments in the generated Python code.

Configuration Keys

KeyTypeDescription
_retry_max_attemptsintMaximum number of retry attempts (including the initial run)
_retry_initial_intervalint (seconds)Delay before the first retry
_retry_backoff_coefficientfloatMultiplier applied to the interval on each subsequent retry
_retry_max_intervalint (seconds)Maximum delay between retries regardless of backoff
_start_to_close_timeoutint (seconds)Max time for a single activity attempt
_schedule_to_close_timeoutint (seconds)Max time from scheduling to completion (across all retries)

Supported Node Types

Error handling config is available on activity_operation nodes and primitive http_request / browse nodes.

Example

Setting retry and timeout config on a graph node:

node config (JSON)
{
  "url": "https://api.example.com/data",
  "method": "GET",
  "_retry_max_attempts": 5,
  "_retry_initial_interval": 2,
  "_retry_backoff_coefficient": 2.0,
  "_retry_max_interval": 60,
  "_start_to_close_timeout": 120
}

This generates a Temporal activity call with:

generated Python
result = await workflow.execute_activity(
    "http_request",
    config,
    start_to_close_timeout=timedelta(seconds=120),
    retry_policy=RetryPolicy(
        maximum_attempts=5,
        initial_interval=timedelta(seconds=2),
        backoff_coefficient=2.0,
        maximum_interval=timedelta(seconds=60),
    ),
)
Tip: If no error handling keys are set, the code generator uses Temporal defaults: 1 attempt with no retry, and a start_to_close_timeout of 60 seconds.

Execution Inspector

The Execution Inspector provides detailed visibility into every activity that ran (or failed) during a workflow execution. It surfaces Temporal's full activity history through the API and renders it in the UI as expandable cards with timing breakdowns.

API

The run status endpoint returns an activity_history array alongside the existing run fields:

GET/execution/runs/{run_id}/status

Activity history fields

FieldTypeDescription
activity_namestringName of the activity that was executed
statusstringscheduled, started, completed, or failed
scheduled_atdatetimeWhen Temporal scheduled the activity
started_atdatetimeWhen the worker picked up the activity
ended_atdatetimeWhen the activity completed or failed
attemptintRetry attempt number (1-based)
inputobjectThe config dict passed to the activity
outputobjectThe result dict returned by the activity (if successful)
errorstringError message (if failed)
error_typestringException class name (if failed)
stack_tracestringFull Python stack trace (if failed)
retry_statestringTemporal retry state, e.g. RETRY_STATE_IN_PROGRESS

UI

The run detail page in the web UI renders each activity as an expandable card showing:

  • Status badge — color-coded (green for completed, red for failed, yellow for in-progress)
  • Timing breakdown — wait time (scheduled to started) and execution time (started to ended)
  • Input / Output viewers — collapsible JSON viewers for the activity's config and result
  • Error details — error message, exception type, and full stack trace for failed activities
  • Retry info — attempt number and retry state

Git Repo Sync

Each workspace can be linked to a git repository. When you sync, Tapcraft:

  1. Shallow-clones (or fast-forward pulls) the repo into workspace_{id}/repo/
  2. Scans repo/activities/ and repo/workflows/ for decorated functions and classes
  3. Registers discovered items so the worker can load them

Configure the workspace via the UI or MCP:

  • repo_url — the git clone URL
  • repo_branch — branch to track (default: main)
  • repo_auth_secret — name of a Tapcraft secret containing a personal access token (for private repos)

Expected repo structure

your-repo/
activities/
  __init__.py
  fetch_data.py         # @activity.defn functions
  send_notification.py
workflows/
  __init__.py
  daily_pipeline.py     # @workflow.defn classes
  monitor.py
Monorepo support: Tapcraft also checks one level of subdirectories — so my-app/activities/ and my-app/workflows/ are also discovered.

MCP Setup

The MCP server lets Claude Code (or any MCP-compatible agent) manage the full Tapcraft lifecycle without touching the UI.

Running the MCP server

terminal

Add the server to your Claude Code MCP config:

~/.claude/claude_desktop_config.json

  "mcpServers": {
    "tapcraft": {
      "command": "python",

      "env": {
        "DATABASE_URL": "sqlite+aiosqlite:///./data/tapcraft.db",

terminal
$ curl -H "X-API-Key: your-key-here" http://localhost:8001/workflows

Or via query parameter: ?api_key=your-key-here

Public endpoints: GET /health returns Temporal connection status. GET /auth/validate checks if a stored API key is valid.

Deployment

Docker Compose (default)

The included docker-compose.yml runs three services with resource limits (1 GB memory, 1 CPU each):

  • temporal — Temporal dev server with in-memory storage
  • api — FastAPI with uvicorn, hot-reload enabled
  • worker — Temporal worker with auto-restart
Note: The default docker-compose.yml uses temporal start-dev which stores data in memory. Temporal state is lost on restart. For production, use a proper Temporal deployment with PostgreSQL/MySQL persistence.

Production considerations

  • TLS — Run behind a reverse proxy (nginx, Caddy, Traefik) that terminates TLS.
  • Temporal — Deploy a production Temporal cluster with persistent storage. See Temporal self-hosted guide.
  • Database — SQLite works for single-node. For multi-node, swap to PostgreSQL via DATABASE_URL.
  • Secrets — Set TAPCRAFT_API_KEY and TAPCRAFT_SECRET_KEY explicitly. Do not rely on auto-generation.
  • Network — Restrict access to the Temporal UI port (8233) to trusted networks.

Request flow

  1. An agent (or the UI) calls the API Server via REST or MCP.
  2. The API writes metadata to SQLite and starts a workflow via Temporal's gRPC API.
  3. Temporal dispatches tasks to the Worker.
  4. The Worker dynamically imports workflow and activity code from the Workspace.
  5. Activities execute I/O (HTTP calls, file operations, etc.) and return results.
  6. Temporal persists execution state, handles retries, and records history.
